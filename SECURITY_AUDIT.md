## 二、已修复漏洞详情

### 🔴 [CRIT-01] Newtonsoft.Json TypeNameHandling RCE

**位置**:
- [游戏服务器/游戏服务器/模板类/序列化类.cs:24](游戏服务器/游戏服务器/模板类/序列化类.cs#L24)
- [账号服务器/序列化类.cs:18](账号服务器/序列化类.cs#L18)

**风险**: `TypeNameHandling = TypeNameHandling.Auto` 允许 JSON 中的 `$type` 字段指定任意 .NET 类型。结合 `JsonConvert.DeserializeObject` 加载来自磁盘文件（账号文件、游戏数据）的内容，攻击者只要能写入数据目录就能通过反序列化 gadget 链获得 RCE。已知 gadget 包括 `System.Windows.Data.ObjectDataProvider`、`System.Configuration.Install.AssemblyInstaller` 等。

**修复**: 引入 `ISerializationBinder` 实现（`安全类型绑定器`），白名单仅允许 `游戏服务器` / `账号服务器` / `Assembly-CSharp` 三个程序集的类型反序列化，其他全部抛 `SerializationException`。`TypeNameHandling.Auto` 保留以维持多态序列化（技能任务子类等），但攻击面被切断。

### 🔴 [CRIT-02] 门票生成使用 System.Random（可预测）

**位置**: [账号服务器/账号数据.cs:7,35](账号服务器/账号数据.cs#L7)

**风险**: `System.Random` 用系统时钟做种子，输出可预测。32 字符的会话门票（`ULS21-XXXXXXXXXX...`）名义上有 ~190 比特熵，但实际只有 ~32 比特种子熵。攻击者只要知道账号服务器启动时间区间，就能脱机暴力枚举 token，劫持任意账号。

**修复**: 替换为 `System.Security.Cryptography.RandomNumberGenerator.Create()`，从内核 CSPRNG 取真随机字节。

### 🔴 [CRIT-03] 账号文件写入路径穿越

**位置**: [账号服务器/主窗口.cs:113](账号服务器/主窗口.cs#L113)

**风险**: `File.WriteAllText(数据目录 + "\\" + 账号.账号名字 + ".txt", ...)`。`账号名字` 来自客户端注册请求，未经任何过滤。攻击者注册账号名 `..\..\..\Windows\System32\drivers\etc\hosts`（或类似）即可向任意路径写入 JSON 序列化数据，导致系统文件被破坏或远程代码执行（写入 Startup 目录、计划任务等）。

**修复**: 增加 `是合法账号名()` 白名单校验（仅允许字母、数字、`_`、`-`，长度 ≤ 64），并使用 `Path.GetFullPath` 做规范化 + 前缀校验，确保最终路径仍在 `数据目录` 之内。

### 🔴 [CRIT-04] 怪物爆率文件路径穿越

**位置**: [游戏服务器/游戏服务器/模板类/游戏怪物.cs:330](游戏服务器/游戏服务器/模板类/游戏怪物.cs#L330)

**风险**: 怪物配置文件中 `array[1]` 可包含 `../` 序列，导致 `File.OpenText` 越权读取任意文件。如果配置文件本身可被低权限角色或外部数据源写入，则攻击者可读取服务器密钥、日志等敏感文件。

**修复**: 增加字符白名单 + `Path.GetFullPath` 前缀校验。

### 🟠 [HIGH-01] TCP 监听 IP 封禁绕过

**位置**: [游戏服务器/游戏服务器/网络类/网络服务网关.cs:395-399](游戏服务器/游戏服务器/网络类/网络服务网关.cs#L395)

**风险**: 被封 IP 的连接虽然调用了 `Close()`，但**紧接着仍然被 `Enqueue` 加入待处理队列**。这是逻辑错误（缺少 `else`），导致 IP 封禁完全失效。

**修复**: 改为 `if/else` 结构。注意没有直接用 `return`，因为函数末尾还有 `BeginAcceptTcpClient` 重新挂载监听器，提前 return 会导致服务器停止接受新连接。

### 🟠 [HIGH-02] 封包字节字段无上界（DoS / 协议越界）

**位置**: [游戏服务器/游戏服务器/网络类/游戏封包.cs:67-68](游戏服务器/游戏服务器/网络类/游戏封包.cs#L67)

**风险**: 当 `描述符.长度 == 0` 时，长度从流中读取 `UInt16`（客户端控制，最大 65535）。`ReadBytes(n)` 会在堆上预先分配 `n` 字节。配合连接并发，可批量制造内存压力；同时还可能导致越界读取下一个封包的数据，造成协议解析错乱。

**修复**: 增加双重校验 —— ① 上限不超过 64KB；② 不超过流剩余可读字节数。任一越界直接抛异常。

### 🟠 [HIGH-03] 弱 TLS 协议版本

**位置**: [游戏服务器/游戏服务器/SMain.cs:93](游戏服务器/游戏服务器/SMain.cs#L93)

**风险**: 启用了 `Tls`（即 TLS 1.0，BEAST 攻击）和 `Tls11`（POODLE 攻击）。这两个版本已被 RFC 8996 正式废弃。

**修复**: 改为 `Tls12 | Tls13`。

以下问题需要客户端配合或大规模架构改造，不在本次自动修复范围内。建议项目方按优先级排期处理。

### 🔴 [PROTO-01] 明文密码存储与比对

**位置**:
- [账号服务器/账号数据.cs:22](账号服务器/账号数据.cs#L22) — `public string 账号密码;`
- [账号服务器/网络通信.cs:89,174](账号服务器/网络通信.cs#L89) — `array[3] != value.账号密码` 直接字符串比对
- [游戏服务器/游戏服务器/地图类/玩家实例.cs:21394](游戏服务器/游戏服务器/地图类/玩家实例.cs#L21394) — 二级密码同样明文

**风险**: 账号密码以明文存储在 `.txt` 文件中，登录时直接字符串比较（同时有时序攻击风险）。账号文件泄露 = 全部用户凭据泄露。

**修复方向**:
1. 引入密码哈希：使用 PBKDF2（.NET Framework 自带 `Rfc2898DeriveBytes`）、bcrypt 或 Argon2id
2. 加盐：每个账号独立随机盐
3. 使用 `CryptographicOperations.FixedTimeEquals` 做常量时间比较
4. 迁移路径：登录时检测旧明文格式 → 升级为哈希 → 重写文件

### 🔴 [PROTO-02] 密码明文经 UDP 传输

**位置**: [游戏登录器/登录界面.cs:224](游戏登录器/登录界面.cs#L224)，[账号服务器/网络通信.cs:29](账号服务器/网络通信.cs#L29)

**风险**: 登录器通过 UDP 明文发送 `账号 密码`。攻击者抓包即得凭据，存在中间人攻击。

**修复方向**:
1. 短期：在客户端和账号服务器之间增加共享对称密钥的简单握手 + AES-GCM 加密
2. 长期：迁移到 TCP + TLS / DTLS

### 🔴 [PROTO-03] 账号服务器 → 游戏服务器之间无身份认证

**位置**: [游戏服务器/游戏服务器/网络类/Http门票接收器.cs:73](游戏服务器/游戏服务器/网络类/Http门票接收器.cs#L73)

**风险**: 原本有的 IP 白名单校验被注释掉。游戏服务器接收 UDP 6678 端口的门票数据**未做任何来源验证**。任何能访问 6678 端口的客户端可注入伪造门票 `<ticket>;<account>` → 直接以任意账号登录，无需密码。

**修复方向**:
1. 取消注释 IP 白名单校验，且 IP 必须从配置文件读取
2. 增加 HMAC-SHA256：账号服务器和游戏服务器配置共享密钥，门票格式改为 `<ticket>;<account>;<timestamp>;<hmac>`
3. 防止时间戳重放（5 分钟有效窗口 + nonce 去重）

### 🔴 [PROTO-04] 多个未鉴权的封包处理器

**位置**: [游戏服务器/游戏服务器/网络类/客户网络.cs:553-688](游戏服务器/游戏服务器/网络类/客户网络.cs#L553)

**风险**: 至少 8 个封包处理器（传奇之力激活、战功信息、战功任务、战功奖励、战功军令、战功积分、主题礼包、获取真实 IP）**没有检查认证状态**（"阶段" 字段）。未登录的客户端可触发这些处理器，部分还会对 `this.绑定角色` 解引用 → 空指针崩溃 / DoS。

**修复方向**:
1. 在每个 `处理封包` 入口统一加 `阶段` 检查（要求 ≥ 已登录）
2. 使用 `[需要登录]` 自定义 attribute + 反射统一拦截，避免每个 handler 单独写
3. 处理 `this.绑定角色?.` 做 null 判断

### 🔴 [DEEP-01] 休眠后门：硬件指纹外发 + 远程控制信道

**位置**: [游戏服务器/游戏服务器/地图类/玩家实例.cs:4010](游戏服务器/游戏服务器/地图类/玩家实例.cs#L4010)（原 `最优恢复` 方法）

**风险**: 该方法用 byte 数组解码字符串的方式**故意规避静态分析**：
- 解码出 `Win32_Processor` 字符串 → WMI 查询 CPU 硬件信息
- 解码出 `base/Uniqueverify` 路径，POST 硬件指纹到 `主程.LogWebSite + base/Uniqueverify`
- 远端可控制 `玩家实例.恢复量`（游戏机制）+ 主界面状态栏文本
- 触发条件：在线玩家 ≥ 5 时静默激活；激活后随机延迟 1000-3000 秒应用响应

`主程.LogWebSite` 默认为 null（赋值代码 `//主程.LogWebSite = "http://58.87.104.125:8081/";` 已在源码中被注释），所以**当前不会触发**。但只要任何路径设置了这个变量，整套外发逻辑立即激活。这是一条**远程控制 / 数据采集信道**。

**修复**: `最优恢复` 方法体清空，仅设置 `下次恢复` 时间和 `恢复量 = -1`。原始实现保留为 `最优恢复_已禁用_原始实现` 供审计参考，永不被调用（首行 `if (true) return;`）。

### 🔴 [DEEP-02] HTTP 充值/管理接口签名形同虚设

**位置**: [游戏服务器/----/_0015_0003_0007_000E_000D_000D.cs:172](游戏服务器/----/_0015_0003_0007_000E_000D_000D.cs#L172)（`Sign` 方法）

**风险**: 原签名算法是 `MD5(query_string + "&")`。"&" 是**硬编码常量**，任何看到源码的人都能算出有效签名。该签名保护以下 HTTP 端点：
- `/useCmd` —— 执行 GM 命令（当前对应的 `主程.AddWebEvent` 调用已注释，但端点本体可被重新激活）
- `/paymentcallback` —— 充值回调（同上）
- `/modifyRole` —— 改角色数据（同上）
- `/getRoleList` —— **批量导出全部角色数据（账号、名字、等级、战力、金币、银币、元宝），未注释，可被立即利用**

攻击者拿到源码即可构造任意签名，调用 `/getRoleList` 一次性拖走全服角色清单。

**修复**:
1. `Settings.cs` 新增 `充值签名密钥` 配置项，默认空字符串
2. `Sign()` 改用 `HMAC-SHA256(query_string, 充值签名密钥)`
3. 密钥为空时 `Sign()` 返回固定哨兵 `__signing_key_not_configured__`，与任何客户端签名都不可能相等 → 所有签名请求直接被拒
4. 签名比对改用 `CryptographicOperations.FixedTimeEquals` 防时序攻击

**部署提示**: 公开发布时 `充值签名密钥` 必须为空（默认安全）。运营方部署时需手动配置一个高熵随机字符串，且对外 HTTP 客户端要同步用同样的密钥做 HMAC-SHA256 签名。

### 🟡 [DEEP-07] WebApi 三处加固

**位置**：[WebApi.cs:OnPostRequest](游戏服务器/网络类/WebApi.cs)

1. **HTTP body 无上限**：`new StreamReader(request.InputStream).ReadToEnd()` 直接读完客户端 body。攻击者用 Content-Length 极大的 POST 可撑爆内存。

1. `发货.cs:31`: `元宝数量` 上限收紧为 `uint.MaxValue / 100u` (≈42M, 对齐 `赠送元宝`)
2. `发货.cs:55` 与 `赠送元宝.cs:52`: 累加溢出校验改用 `(ulong)元宝数量.V + num > int.MaxValue`, 防 uint+uint 回绕

### 已知风险但不在本次范围 (post-compromise)

下列属于"需要先攻陷服务器或账号"才能利用的二线问题, 按既定原则不在本次自动修复:
